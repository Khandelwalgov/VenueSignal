# Accessibility

Accessibility is both product logic and interface quality. The deterministic engine verifies step-free routes and permits a no-route answer. The interface provides a skip link, semantic headings, labelled controls, mirrored map lists, SVG title/description, keyboard-selectable entities, visible focus, non-colour symbols, text route status, loading/error status, responsive layouts, and reduced-motion handling.

The incident workflow uses native labels, file input, buttons, checkboxes, semantic `h2`/`h3` structure, and live textual results. Route state, context-version changes, plan validity, approvals, and task/communication transitions use textual status announcements. The no-route state is uppercase, symbol-and-text based, and does not depend on colour. Task and communication queues remain keyboard operable; translated content carries language attributes; publication is explicitly labelled simulated. The skip link targets a programmatically focusable main region.

The final browser pass found no duplicate IDs, no heading-level jumps, no unnamed interactive controls in the accessibility tree, and no document-level horizontal overflow at 390 px, 768 px, or the 640-CSS-pixel reflow equivalent of a 1280 px viewport at 200% zoom. Reduced-motion rules are present. Native controls and mirrored lists provide the complete keyboard path with no dialogs or keyboard traps. A production launch still requires current VoiceOver/NVDA/JAWS and automated WCAG-tool verification on the deployed build; that external assistive-technology audit was not claimed locally.
